Language Design

Key Features

  • Value pipes
  • First-class channels (= pipes)
  • Streaming functions: output, not return
  • Asynchronous composition: ... | ...
  • Synchronous composition: (...)
  • Unifies both types of output: put

High-level Example

lexical ::= %x              access ::= $obj!slot
dynamic ::= $x              vector ::= [ ... ]
binder ::= ?x               lambda ::= ( ... => ... )
bare ::= x                  funcdef ::= (f ...) = ...

# given a filename that contains a list of urls,
# return all the titles and descriptions.
(get-titles-and-descriptions ?f) =
  file-lines $f
    | fan $threads %get-webpage
    | each (?page =>
        put [($page!select title)
             ($page!select description)])
    | each [%map %downcase]

Concurrency Primitives

Channels as values

channel = (make-channel)

# write values with `put`
put 1 2 3 > $channel

# read a value with `get`
get < $channel

Spawning: Run in the background

& f arg arg
& f >$out <$in
& foo | bar | baz

Anonymous channels (pipes)

command-1 | command-2

# equivalent to

channel = (make-channel)

& command-1 > $channel
command-2 < $channel

Concurrency: Interruption

Why is Interruption Necessary?

Pattern: Full Iteration

file-lines $f
  | each (=> ...)

Implementation: Infinite Loops

(each ?fn) =
  loop (=> fn (get))
Problem: How does each know when to stop?

Pattern: Partial Iteration

file-lines $f
  | take 10

Implementation: Simple Return

(take ?n) = gt 0 %n &&
  (get; take (decr %n))
Problem: How does file-lines know when to close the file?

Concurrency: Interruption

Idea: Probably blocked forever → Interrupt!

Concurrency: Interruption

When to close a channel?

when the last reader exits

No more writes possible → close!

when the last writer exits

No more reads possible → close!

When to interrupt a process?

Currently waiting channel closes:

Probably blocked forever → interrupt!

Try to read/write a closed channel:

Probably blocked forever → interrupt!

“The last one to leave has to turn off the lights”

UNIX integration

  • Incomplete
  • Why? Needed to solve the input-stealing problem.

    - UNIX commands assume exclusive read access

    - This can make inputs disappear!

  • Solution designed, but not implemented.

    - Syntax-driven detection of when stdin is used

Validation

Need to validate:

  • Magritte is appropriate for writing large programs

    problem: Lots of concurrent code, which is difficult to debug

    BUT: Have enabled programs which use data structures, not possible in bash

  • Magritte is appropriate for shell scripting

    problem: UNIX integration is more difficult than we first expected

  • Magritte is appropriate as an interactive system shell

    problem: Also needs UNIX integration

    BUT: Have preserved the shell composition strategy and repl interaction
